# SmartHire 招聘与面试管理系统开发计划 V2

## 1. 项目定位

**SmartHire** 是一个基于 Java 后端的招聘与面试管理系统，目标不是堆很多功能，而是做出一个：

- 可以写进简历的 Java 项目
- 可以在面试中完整讲清楚的后端项目
- 能体现业务建模、权限设计、工程化能力的项目

本项目的核心定位是：

1. **优先完成真实业务闭环**
2. **优先保证后端完整性**
3. **优先做能讲清楚的设计，而不是做得很杂**

---

## 2. V2 成功标准

如果项目完成到以下程度，就已经是一个合格的简历项目：

- 候选人可以注册、登录、浏览岗位、搜索岗位、投递简历、查看投递记录
- HR 可以发布岗位、管理岗位、查看投递列表、推进申请状态、安排面试
- 系统支持 JWT 鉴权与基础 RBAC 权限控制
- 系统支持通知记录，形成完整招聘流程闭环
- 仓库包含 README、SQL 初始化脚本、接口文档说明，并能口头清楚讲解核心业务与工程设计

一句话概括：**至少要能完整演示一次“注册 -> 登录 -> 浏览岗位 -> 投递 -> HR 处理 -> 安排面试 -> 候选人查看通知”的主流程。**

## 4. 角色设计

系统包含 3 类角色，但 V2 坚持**固定角色、轻量后台**，不做复杂的动态权限平台。

### 4.1 候选人（Candidate）
- 注册、登录
- 浏览岗位
- 搜索岗位
- 投递岗位
- 查看投递记录
- 查看面试安排
- 查看通知消息

### 4.2 HR
- 发布岗位
- 管理岗位
- 查看候选人投递列表
- 更新申请状态
- 安排面试
- 查看基础招聘统计

### 4.3 管理员（Admin）
- 用户状态管理
- 岗位状态管理
- 查看操作日志或系统概览

> V2 不建议做“动态角色管理”后台。这个功能实现成本高，但对简历价值不如业务闭环、权限控制和状态流转。

- 防重复投递

#### 面试管理
- HR 安排面试
- 候选人查看面试时间和地点
- 更新面试结果

#### 通知模块
- 投递成功通知
- 面试安排通知
- 状态更新通知
- 站内通知查看与已读管理

### 5.2 P1：应该完成

这部分会明显提升项目完整度，但不应阻塞 P0。

- 简历 PDF 上传
- 基础数据统计
- 管理员轻量后台
- Docker 化部署
- README 与接口文档完善
- 基础操作日志

### 5.3 P2：有时间再做

这部分属于工程亮点，不应影响主流程交付。

- Redis 缓存热门岗位、岗位详情与统计概览（已完成第一阶段）
- RabbitMQ 异步通知（已完成第一阶段）
- 定时任务关闭过期岗位或发送提醒
- 更细的统计图表
- CI/CD 或自动化部署校验

### 5.4 当前明确不做

为了控制范围，V2 默认不做以下内容：

- 复杂动态角色权限管理平台
- 多轮面试流程编排
- 邮件 / 短信真实发送
- 大而全的后台管理系统
- 过度复杂的前端 UI

## 6. 数据库设计建议

V2 建议在项目一开始就把核心实体设计清楚，哪怕部分功能后面再实现。

### 6.1 核心表
- `user`
- `role`
- `user_role`
- `job`
- `application`
- `interview`
- `notification`

### 6.2 可选增强表
- `operation_log`
- `resume_file` 或在 `user` / `application` 中存简历路径

### 6.3 关键设计点

- `application` 表建议对 `candidate_id + job_id` 建唯一约束，用于防重复投递
- `job` 应包含创建人、状态、截止时间、城市、分类等字段
- `application` 需要明确状态流转：`APPLIED -> REVIEWING -> INTERVIEW -> OFFERED / REJECTED`
- `interview` 建议先按单轮面试设计，避免一开始把流程做复杂
- `notification` 应包含消息类型、接收人、是否已读、关联业务 ID

> 重点不是表越多越好，而是你能解释清楚每张表为什么存在、状态为什么这样流转。

## 7. 接口与模块边界建议

为了后续更容易讲清楚项目，建议从一开始就把接口边界想清楚。

### 7.1 认证相关
- `POST /auth/register`
- `POST /auth/login`

### 7.2 岗位相关
- `POST /jobs`
- `PUT /jobs/{id}`
- `DELETE /jobs/{id}`
- `GET /jobs/{id}`
- `GET /jobs`

### 7.3 投递相关
- `POST /applications`
- `GET /applications/me`
- `GET /jobs/{id}/applications`
- `PUT /applications/{id}/status`

### 7.4 面试相关
- `POST /interviews`
- `PUT /interviews/{id}`
- `GET /interviews/me`

### 7.5 通知相关
- `GET /notifications`
- `PUT /notifications/{id}/read`

> 不要求一开始就把所有接口写满，但建议先把资源边界设计出来，后面联调和文档都会更顺。

## 8. 开发路线图（建议改为阶段制）

V2 不建议再按“每天必须完成什么”硬排，因为这样容易在某个模块卡住后整体失速。更现实的方式是**按阶段推进**。

## 第 1 阶段：基础骨架与权限（3~5 天）

目标：把项目跑起来，并完成认证与基础权限。

- 创建 Spring Boot 项目
- 配置 Maven、MySQL、基础目录结构
- 设计核心表结构与 SQL 脚本
- 完成注册、登录、密码加密
- 接入 JWT
- 完成基础 RBAC
- 补充统一返回格式、全局异常处理、Swagger

**阶段产出：**
- 可启动的后端项目
- 注册 / 登录可用
- 基础权限跑通
- SQL 初始化脚本初版

## 第 2 阶段：核心业务闭环（4~6 天）

目标：完成最重要的招聘主流程。

- 岗位 CRUD
- 岗位搜索与分页
- 候选人投递岗位
- 防重复投递
- HR 查看投递列表
- 申请状态流转
- HR 安排面试
- 候选人查看面试安排
- 站内通知生成与查看

**阶段产出：**
- 招聘主流程可以完整演示
- 系统已经具备“简历可写”的基本价值

## 第 3 阶段：项目包装与稳固（3~5 天）

目标：补足文档、测试、部署与轻量增强。

- README 完善
- 补充接口文档
- 核心流程自测
- 增加关键测试用例
- 管理员轻量功能
- 简历上传
- Docker 化部署

**阶段产出：**
- GitHub 仓库结构完整
- 项目可本地启动和演示
- 面试时有完整讲解材料

## 第 4 阶段：工程亮点增强（可选）

只有在前 3 个阶段都稳定完成后，再考虑以下内容：

- Redis 缓存
- RabbitMQ 异步通知
- 定时任务
- 操作日志增强
- 更细的统计图表
- CI/CD

**阶段产出：**
- 项目具备工程化亮点
- 能体现你对性能、解耦、部署的理解

## 9. 最少要补的测试

V2 不建议只停留在 Postman 手测。至少应覆盖以下场景：

- 注册 / 登录成功与失败
- 未登录访问受保护接口
- 不同角色访问权限校验
- 重复投递是否被拦截
- 申请状态是否按规则流转
- 面试安排后通知是否生成

> 如果时间有限，优先补集成测试或关键业务测试，而不是追求测试数量。

## 12. 面试讲解重点

项目完成后，至少应能清楚回答以下问题：

- 为什么选择招聘系统而不是更常见的博客 / 商城题目
- 数据库为什么这样设计
- `application` 为什么要做唯一约束
- JWT 登录认证流程是什么
- Spring Security 在项目里负责什么
- 申请状态为什么这样设计
- 如何保证不同角色只能访问对应接口
- 为什么通知先做同步，再考虑异步
- 如果继续迭代，下一步会优先优化什么

## 13. 开发原则

本项目遵循以下原则：

1. **先做业务闭环，再做技术增强**
2. **先保证后端完整，再补前端体验**
3. **先做 P0，再做 P1，最后再考虑 P2**
4. **优先做面试可讲的功能，而不是堆技术名词**
5. **每新增一个功能，都要回答“它是否提高了项目的可解释性”**
