# SmartHire Interview Questions

这份文档用于整理项目相关的高频面试题。当前已收录 `Redis`、`JWT`、`Docker / Docker Compose`、`RabbitMQ`、`RBAC`、`Spring Security`、`MyBatis-Plus`、`MySQL / 数据库设计`、`事务与一致性` 相关问题，后续可以继续补测试策略、前端认证与部署等模块。

## Redis

### 1. Redis 是什么？

Redis 是一个基于内存的键值数据库，读写速度很快，常用于缓存、计数器、排行榜、分布式锁、会话存储和消息队列等场景。

一句话可以这样讲：

`Redis 不是用来替代 MySQL 主业务数据库的，而是用来承接高频读写、降低数据库压力、提升响应速度的内存型中间件。`

### 2. 为什么项目里要用 Redis？

在 SmartHire 里，最适合先上 Redis 的场景是读多写少、重复访问高的接口：

- 公共岗位列表
- 岗位详情
- 统计概览

可以这样回答：

`我把 Redis 先落在公共岗位列表、岗位详情和统计概览上，因为这些接口读多写少、重复访问高，适合缓存。这样可以减少 MySQL 压力，提升接口响应速度。`

### 3. Redis 的优势是什么？

- 基于内存，读写非常快
- 适合高频访问数据缓存
- 支持过期时间，天然适合缓存场景
- 数据结构丰富，不只是字符串
- 支持原子操作，适合计数和简单并发控制
- Java 生态成熟，和 Spring Boot 集成方便

### 4. Redis 的缺点是什么？

- 内存成本高，比磁盘贵
- 容量有限，不适合存全量业务数据
- 会带来缓存与数据库的一致性问题
- 引入了额外的部署、监控和运维成本
- 如果没有降级策略，Redis 出问题可能影响业务

### 5. Redis 和 MySQL 的区别是什么？

可以这样讲：

- MySQL 是主业务数据库，负责持久化存储和事务一致性
- Redis 更适合做高频访问数据的缓存或辅助中间件
- MySQL 重在“准”和“持久”
- Redis 重在“快”和“减压”

一句话版：

`MySQL 负责真数据，Redis 负责热数据。`

### 6. 为什么 Redis 适合做缓存？

缓存最核心的诉求通常是：

- 读取快
- 支持过期
- 可以接受短时间最终一致

Redis 正好满足这些特点，所以非常适合放在数据库前面做缓存层。

### 7. 你这个项目里 Redis 是怎么设计的？

当前项目里的第一阶段设计是：

- 缓存对象
  - 公共岗位列表
  - 岗位详情
  - 统计概览
- 缓存 key
  - 岗位列表：按分页和筛选条件拼接 key
  - 岗位详情：按 `jobId`
  - 统计概览：按角色范围区分，`ADMIN` 看全局，`HR` 按用户范围
- 缓存失效
  - 岗位新增 / 更新 / 删除后清理岗位缓存和统计缓存
  - 投递创建 / 状态更新后清理统计缓存
  - 面试安排 / 更新后清理统计缓存

可以这样总结：

`我没有把 Redis 随便铺在所有接口上，而是优先缓存公共高频读接口，并在关键写操作后做缓存失效，平衡性能和一致性。`

### 8. 为什么不是所有接口都缓存？

因为不是所有接口都适合缓存。

例如：

- 候选人自己的通知、投递、面试属于个性化数据
- 这些数据更新频率更高
- 缓存收益未必高，反而会增加复杂度和一致性成本

所以优先缓存“公共、高频、读多写少”的接口更合理。

### 9. 会不会有脏数据？

会，缓存系统天然就可能出现短时间不一致。

可以这样回答：

`会有短暂不一致的可能，所以我在关键写操作后做了缓存失效，而不是只依赖 TTL 慢慢过期。这样可以尽量缩小脏数据窗口。`

### 10. 什么是缓存失效？

缓存失效就是业务数据变化后，把旧缓存删掉，让下一次读取重新查询数据库并回填缓存。

在当前项目里：

- 写操作发生后执行 `evict`
- 下一次查询再从数据库加载并重新写入缓存

### 11. 什么是缓存穿透、缓存击穿、缓存雪崩？

这是 Redis 面试高频题。

#### 缓存穿透

查询一个数据库里本来就不存在的数据，缓存里也没有，结果每次请求都打到数据库。

#### 缓存击穿

某个热点 key 过期瞬间，大量请求同时打到数据库。

#### 缓存雪崩

大量 key 在同一时间失效，导致数据库压力瞬间暴涨。

当前项目如果继续增强，可以考虑：

- 空值缓存
- 热点 key 防重建
- 随机过期时间
- 更细的缓存分层策略

### 12. 为什么没有直接把 Redis 当数据库？

因为招聘系统这种业务更依赖关系建模、一致性和持久化能力，所以主存储应该还是 MySQL。

Redis 更适合承担：

- 缓存层
- 热点数据层
- 辅助能力层

而不是作为这个项目的主业务数据库。

### 13. 如果 Redis 挂了怎么办？

这是很关键的面试问题。

可以这样回答：

`我在缓存层加了容错处理，Redis 不可用时，业务可以回退到数据库查询，不会因为缓存层不可用导致主流程直接失败，代价只是性能下降。`

这说明你把 Redis 当成“性能增强层”，而不是“业务唯一依赖层”。

### 14. 这个项目里关于 Redis 的一句话总结怎么讲？

可以直接这样说：

`我在这个项目里把 Redis 作为缓存层，而不是主存储层，优先缓存了公共岗位列表、岗位详情和统计概览，通过写操作后的缓存失效来平衡性能和一致性，同时保留了 Redis 不可用时回退数据库的降级能力。`

## JWT

### 1. JWT 是什么？

JWT 全称是 `JSON Web Token`，是一种基于 JSON 的令牌格式，常用于前后端分离场景下的身份认证。

它通常由三部分组成：

- Header
- Payload
- Signature

可以这样讲：

`JWT 本质上是服务端签发给客户端的一段令牌，客户端后续请求带着它，服务端通过校验签名来确认身份。`

### 2. 为什么项目里用 JWT，而不是 Session？

因为这个项目是前后端分离架构，更适合无状态认证。

可以这样回答：

- Session 方案需要服务端维护会话状态
- JWT 更适合前后端分离和 REST API
- 后端不需要额外存用户会话，扩展性更好
- 前端只要在请求头里携带 `Bearer Token` 即可

一句话版：

`我选 JWT 是因为它更适合前后端分离、无状态认证和后续的服务扩展。`

### 3. JWT 的结构是什么？

JWT 由三段组成，用 `.` 分隔：

- Header：声明算法和类型
- Payload：放业务声明数据，比如用户 id、邮箱、角色
- Signature：对前两部分签名，防止篡改

注意点：

- Payload 是可解码的，不是加密的
- 不应该把密码这类敏感信息放进去

### 4. 你这个项目里 JWT 是怎么用的？

当前项目里的 JWT 用法是：

- 登录成功后签发 access token
- 前端后续请求通过 `Authorization: Bearer <token>` 发送
- 后端过滤器解析 token，恢复当前登录用户
- 再把用户信息交给 Spring Security 做鉴权和角色判断

项目里 token 主要携带的信息包括：

- `userId`
- `email`
- `fullName`
- `roles`

### 5. JWT 的优势是什么？

- 无状态，服务端不需要保存会话
- 适合前后端分离和移动端场景
- 可以天然放在网关或微服务认证链路里
- 跨服务传递用户身份更方便

### 6. JWT 的缺点是什么？

- 一旦签发出去，天然不容易主动失效
- 如果泄漏，在过期前可能被滥用
- 令牌太大时，会增加请求头开销
- 需要注意前端存储方式和过期处理

### 7. JWT 和 Session 的核心区别是什么？

可以这样答：

- Session：服务端保存登录状态，客户端通常只存 session id
- JWT：服务端签发 token，客户端带 token，服务端校验签名
- Session 更偏“有状态”
- JWT 更偏“无状态”

一句话总结：

`Session 是服务端记住你，JWT 是客户端拿着凭证来证明自己。`

### 8. JWT 安全吗？

JWT 不是天然绝对安全，关键要看怎么用。

安全使用要点包括：

- 使用足够强的签名密钥
- 配置合理过期时间
- 只放必要声明，不放敏感数据
- 前端避免不安全存储
- 敏感操作可以进一步配合服务端校验

### 9. 为什么说 Payload 不能放敏感信息？

因为 JWT 默认只是签名，不是加密。

别人拿到 token 后，虽然没法伪造签名，但通常能解码看到 payload 里的内容。所以：

- 可以放用户 id、角色、邮箱
- 不应该放密码、身份证号、银行卡号等敏感数据

### 10. JWT 怎么防止被篡改？

靠签名。

服务端会用密钥对 Header 和 Payload 做签名。客户端改了内容后，签名就对不上了，服务端校验时会失败。

### 11. JWT 怎么退出登录？

这是高频问题。

当前这个项目用的是典型简化方案：

- 前端删除本地 token
- 后续请求不再携带 token

你可以主动补一句：

`如果要做更严格的服务端失效控制，可以再配合 Redis 黑名单、短 token + refresh token、或者令牌版本号机制。`

### 12. 为什么项目里没有 refresh token？

因为这个项目当前阶段优先完成主流程和权限闭环，先用 access token 已经够支撑演示和核心业务。

如果继续增强，可以补：

- refresh token
- token 续签
- token 黑名单

### 13. 你这个项目里 JWT 和 Spring Security 是怎么配合的？

可以这样讲：

- 登录时生成 JWT
- 请求进入后由 JWT 过滤器解析 token
- 解析后构造当前用户对象
- 放进 Spring Security 上下文
- 控制器和业务层再基于角色做接口权限与数据归属校验

这也是你项目里 `JWT + RBAC + 数据归属校验` 的核心链路。

### 14. 关于 JWT 的一句话总结怎么讲？

可以直接这样说：

`我在这个项目里用 JWT 实现前后端分离下的无状态认证，登录后由服务端签发 token，请求经由过滤器解析并恢复用户身份，再结合 Spring Security 和角色权限完成接口访问控制。`

## Docker / Docker Compose

### 1. Docker 是什么？

Docker 是一种容器化技术，用来把应用和它的运行环境一起打包，让它在不同机器上以更一致的方式运行。

可以这样讲：

`Docker 的核心价值是把应用运行所需的环境、依赖和配置一起封装起来，减少“我机器上可以跑、你机器上跑不起来”的问题。`

### 2. 容器和虚拟机有什么区别？

可以这样答：

- 虚拟机连操作系统一起虚拟，通常更重
- 容器共享宿主机内核，通常更轻、更快启动
- 虚拟机隔离更完整
- 容器更适合应用部署和开发交付

一句话版：

`虚拟机更像整台电脑，容器更像打包好的应用运行盒子。`

### 3. 为什么项目里要用 Docker？

因为这个项目有多个运行组件：

- MySQL
- Redis
- Backend
- Frontend

如果全靠手动装环境，联调和演示成本会很高。

所以可以这样讲：

`我用 Docker 是为了把数据库、缓存、后端和前端的运行环境标准化，降低本地启动和演示成本。`

### 4. Dockerfile 是什么？

Dockerfile 是构建镜像的说明文件，告诉 Docker：

- 基于什么基础镜像
- 拷贝哪些文件
- 执行哪些构建命令
- 容器启动时运行什么命令

你项目里：

- backend 有自己的 `Dockerfile`
- frontend 也有自己的 `Dockerfile`

### 5. 镜像和容器的区别是什么？

- 镜像是静态模板
- 容器是镜像运行起来之后的实例

一句话版：

`镜像像安装包，容器像启动后的程序实例。`

### 6. Docker Compose 是什么？

Docker Compose 用来编排多个相关容器，一次性定义和启动整套服务。

在这个项目里，它负责一起拉起：

- MySQL
- Redis
- Spring Boot backend
- React frontend

### 7. 为什么项目里要用 Docker Compose？

因为单独启动每个服务太碎了。

你可以这样讲：

`我用 Docker Compose 把多服务启动方式统一起来，开发和演示时只需要一条命令，就能把数据库、缓存、后端和前端全部跑起来。`

### 8. 你这个项目里 Docker Compose 是怎么设计的？

当前设计包括：

- `mysql` 负责主业务数据库
- `redis` 负责缓存层
- `backend` 负责 API 和业务逻辑
- `frontend` 负责独立前端页面
- MySQL 通过挂载 `sql/` 目录自动初始化数据
- 前端通过反向代理把 `/api` 请求转到后端

### 9. Docker Compose 里服务之间怎么通信？

同一个 Compose 网络里的服务可以直接通过服务名互相访问。

比如你项目里：

- backend 连接 MySQL 用 `mysql`
- backend 连接 Redis 用 `redis`

所以环境变量里不是写 `localhost`，而是写容器服务名。

### 10. 为什么前端容器里用了 Nginx？

因为独立前端构建后本质上是静态文件，适合用 Nginx 提供服务。

同时 Nginx 还能顺手做：

- 静态资源分发
- `/api` 反向代理
- 更接近真实部署方式

### 11. Docker 的优势是什么？

- 环境一致性好
- 启动和交付更方便
- 新人接手成本低
- 更容易本地复现和演示
- 对多服务项目尤其友好

### 12. Docker 的缺点是什么？

- 学习和排障成本比本地直跑更高
- 会引入镜像构建、容器网络、卷挂载等额外概念
- 对资源有额外消耗
- 如果配置写得不好，调试体验会变差

### 13. 为什么这个项目里不是只给后端做 Docker？

因为这个项目已经是完整的前后端分离加多服务依赖结构，只给后端做 Docker 不能完整体现交付能力。

现在一起容器化后，项目更像真正可交付的整套系统。

### 14. Docker 和 Docker Compose 的区别是什么？

- Docker 关注单个容器怎么构建和运行
- Docker Compose 关注多个容器怎么协同工作

一句话版：

`Docker 解决“一个服务怎么装进容器”，Compose 解决“一组服务怎么一起跑”。`

### 15. 你这个项目里关于 Docker 的一句话总结怎么讲？

可以这样说：

`我在这个项目里用 Docker 和 Docker Compose 把 MySQL、Redis、后端和前端统一容器化，解决本地环境一致性和多服务联调问题，让项目可以通过一条命令完成启动和演示。`

## RabbitMQ

### 1. RabbitMQ 是什么？

RabbitMQ 是一个消息队列中间件，用来在系统之间传递消息，实现异步处理和业务解耦。

可以这样讲：

`RabbitMQ 的核心作用是把消息发送方和处理方解耦，让主业务先完成，再由消费者异步处理后续动作。`

### 2. 为什么项目里要用 RabbitMQ？

在 SmartHire 里，通知就是最适合异步化的一类操作：

- 候选人投递成功后通知 HR
- HR 更新投递状态后通知候选人
- HR 安排面试后通知候选人
- HR 更新面试信息后通知候选人

可以这样回答：

`我引入 RabbitMQ 的主要原因是把主业务链路和通知链路解耦。这样投递、状态流转、面试安排这些主流程可以先快速提交成功，通知由消息消费者异步处理。`

### 3. RabbitMQ 的核心概念有哪些？

最常见的几个概念是：

- Producer：消息生产者
- Consumer：消息消费者
- Queue：消息队列
- Exchange：交换机，负责路由消息
- Routing Key：路由键，决定消息流向
- Binding：交换机和队列之间的绑定关系

一句话版：

`Producer 发消息到 Exchange，Exchange 按规则把消息路由到 Queue，Consumer 再从 Queue 里取消息处理。`

### 4. Exchange 有哪些类型？

面试里最常见的有四种：

- `Direct`：按精确路由键转发
- `Topic`：按模式匹配路由
- `Fanout`：广播到所有绑定队列
- `Headers`：按消息头匹配

当前项目里更适合用 `Topic`，因为后续通知事件可以按业务类型拆分，例如：

- `application.submitted`
- `application.status.changed`
- `interview.scheduled`
- `interview.updated`

### 5. 为什么不能直接同步调用，而要用 MQ？

因为通知不是主业务链路的核心成功条件。

如果同步做：

- 用户请求会更慢
- 通知失败可能拖垮主流程
- 后续扩展邮件、短信、站内信时耦合会更重

所以可以这样讲：

`我把通知从同步调用改成 RabbitMQ 异步消息，是为了降低主流程响应时间，并让业务主链和通知侧解耦。`

### 6. RabbitMQ 的优势是什么？

- 适合异步处理和业务解耦
- 可以削峰，平滑处理突发消息
- 方便扩展多个消费者
- 支持重试、死信、确认机制
- 和 Spring Boot 集成成熟

### 7. RabbitMQ 的缺点是什么？

- 系统架构会更复杂
- 排障难度比同步调用高
- 需要考虑消息丢失、重复消费、顺序性、幂等等问题
- 会增加部署和运维成本

### 8. RabbitMQ 和 Redis 的区别是什么？

可以这样答：

- Redis 在这个项目里主要承担缓存层角色，解决高频读性能问题
- RabbitMQ 主要承担消息中间件角色，解决异步解耦问题
- Redis 更偏“热数据访问”
- RabbitMQ 更偏“任务和事件流转”

一句话版：

`Redis 解决读性能问题，RabbitMQ 解决异步解耦问题。`

### 9. 什么是 ack？

ack 是消费者处理完成后给 RabbitMQ 的确认。

如果消费者没有成功 ack，消息可能会被重新投递。

所以可以这样讲：

`ack 的作用是告诉 Broker 这条消息已经被成功消费，避免消息在未完成处理时被误删。`

### 10. 为什么会重复消费？怎么处理？

消息队列里重复消费是常见问题，例如：

- 消费者处理完了，但 ack 没发成功
- 网络抖动导致 Broker 认为没处理成功
- 消费者中途崩溃，消息重新投递

解决思路是幂等。

在这个项目里可以这样做：

- 用业务主键控制通知重复生成
- 或者增加消息唯一标识和消费记录
- 消费前先判断该通知是否已经创建过

### 11. 什么是死信队列？

死信队列就是接收“正常流程处理不了的消息”的队列。

常见进入死信队列的场景：

- 消息被拒绝
- 重试多次仍失败
- 消息过期

它的价值在于：

- 不让坏消息反复阻塞主消费队列
- 方便后续人工排查和补偿处理

### 12. 你这个项目里 RabbitMQ 怎么设计比较合理？

当前项目最适合的第一版设计是：

- 一个通知交换机，例如 `smarthire.notification`
- 使用 `topic exchange`
- 几类 routing key：
  - `application.submitted`
  - `application.status.changed`
  - `interview.scheduled`
  - `interview.updated`
- 一个通知消费队列先统一接住这些事件
- 消费者负责生成 `notifications` 表记录

这样设计的好处是：

- 先把通知异步化这条链跑通
- 后面如果要加邮件或短信，只需要扩消费者
- 主业务服务不用关心通知怎么落库

### 13. 这个项目里 RabbitMQ 会落在哪些地方？

可以直接这样讲：

`我会把 RabbitMQ 落在通知链上。投递成功、投递状态变更、面试安排、面试更新这些业务事件在事务提交后发布消息，由通知消费者异步生成站内通知。`

当前最适合异步化的事件包括：

- `APPLICATION_SUBMITTED`
- `APPLICATION_STATUS_CHANGED`
- `INTERVIEW_SCHEDULED`
- `INTERVIEW_UPDATED`

### 14. 如果消息丢了怎么办？

这是高频问题。

常见思路包括：

- 生产端确认消息是否成功投递到 Broker
- 队列和消息都做持久化
- 消费端采用手动 ack
- 失败消息进入死信队列

如果想回答得更稳一点，可以补一句：

`消息可靠性不是单点配置能解决的，而是要从生产、存储、消费三段一起设计。`

### 15. 这个项目里为什么适合先做通知异步化，而不是别的功能？

因为通知天然不是强同步主链。

相比岗位、投递、面试这些核心写操作，通知更适合后移：

- 即使通知稍慢一点，主业务也仍然成立
- 即使通知短暂失败，也可以重试或补偿
- 业务上更容易接受最终一致

所以这块非常适合作为 RabbitMQ 的第一落点。

### 16. 关于 RabbitMQ 的一句话总结怎么讲？

可以直接这样说：

`我在这个项目里把 RabbitMQ 作为异步解耦层，优先把投递、状态流转和面试相关的通知改成消息驱动处理，让主业务链路更轻，通知能力也更容易扩展。`

## RBAC

### 1. RBAC 是什么？

RBAC 全称是 `Role-Based Access Control`，中文一般叫“基于角色的访问控制”。

它的核心思想是：

- 先定义角色
- 再把权限赋给角色
- 用户通过角色获得权限

一句话版：

`RBAC 不是直接给每个用户配权限，而是先定义角色，再通过角色统一管理权限。`

### 2. 为什么项目里要用 RBAC？

因为 SmartHire 天然就是多角色系统：

- `CANDIDATE`
- `HR`
- `ADMIN`

如果不用角色模型，接口权限会很散，后续维护也会很乱。

可以这样讲：

`我用了 RBAC 来统一管理 Candidate、HR 和 Admin 三类用户的权限边界，这样鉴权逻辑更清晰，也更容易扩展。`

### 3. 你这个项目里 RBAC 是怎么落地的？

当前项目的角色模型是：

- `CANDIDATE`：投递、查看自己的申请/面试/通知
- `HR`：发布岗位、查看自己岗位下的投递、安排面试
- `ADMIN`：全局统计、用户管理、岗位总览、操作日志

数据库上对应：

- `roles`
- `user_roles`

认证后会把角色一起放进当前用户上下文，再由 Spring Security 和业务层一起判断。

### 4. 为什么不是直接给用户加一个 role 字段？

如果只有单角色系统，一个 `role` 字段也能跑。

但用 `user_roles` 中间表更灵活，因为它：

- 支持用户多角色
- 结构更符合标准权限模型
- 后面扩展权限体系时改动更小

所以可以这样回答：

`我这里虽然当前主要是单角色使用场景，但还是按更标准的 user-role 关联表建模，方便后续扩展。`

### 5. RBAC 能完全解决权限问题吗？

不能。

RBAC 只能回答“你是什么角色”，但还不能完全回答“你能不能操作这条具体数据”。

例如：

- `HR` 角色可以管理岗位
- 但不是所有 HR 都能改所有岗位
- 还要再校验“这个岗位是不是你自己发布的”

所以你这个项目实际上是：

`RBAC + 数据归属校验`

### 6. 什么是数据归属校验？

就是在角色判断通过后，再判断这条数据是不是当前用户有权操作的那一条。

你项目里典型场景有：

- HR 只能管理自己创建的岗位
- HR 只能处理自己岗位下的投递
- HR 只能安排自己岗位对应申请的面试
- 候选人只能看自己的申请、面试和通知

这会让权限模型更接近真实业务。

### 7. 为什么管理员可以看全局，而 HR 只能看自己范围？

因为角色职责不同。

- `ADMIN` 偏系统管理和全局运营视角
- `HR` 偏自己业务域内的招聘操作

如果让 HR 默认拥有全局数据权限，既不符合业务，也会带来数据越权风险。

### 8. 如果后面要加新角色，怎么扩展？

常见做法是：

- 在 `roles` 表新增角色
- 给用户写入新的 `user_roles`
- 在安全配置和业务校验里补上新角色规则

如果需要更细粒度，还可以再往下扩：

- 角色 -> 权限点
- 权限点 -> 资源动作

但当前项目阶段没必要做那么重。

### 9. 为什么你没有做动态权限后台？

因为这个项目的目标是“讲清楚主流程和工程设计”，不是做一个特别复杂的权限平台。

动态权限后台会带来：

- 菜单权限
- 资源权限
- 动态配置
- 更多管理端复杂度

对当前简历项目来说投入产出比不高。

### 10. 关于 RBAC 的一句话总结怎么讲？

可以直接这样说：

`我在项目里用 RBAC 管理 Candidate、HR、Admin 三类用户的接口访问权限，再结合数据归属校验限制具体资源操作范围，既保证了权限边界清晰，也更贴近真实业务。`

## Spring Security

### 1. Spring Security 是什么？

Spring Security 是 Spring 生态里的安全框架，主要负责：

- 认证
- 鉴权
- 安全上下文管理
- 过滤器链

一句话版：

`Spring Security 是用来管理登录身份、接口权限和请求安全链路的核心框架。`

### 2. 为什么项目里要用 Spring Security？

因为你这个项目有：

- 登录认证
- JWT 解析
- 多角色权限控制
- 公开接口和受保护接口并存

如果全靠手写拦截器和判断，会比较散，也不容易维护。

所以可以这样讲：

`我用 Spring Security 统一管理认证和鉴权，让 JWT、角色权限和接口访问控制都走标准安全链路。`

### 3. 你这个项目里 Spring Security 是怎么用的？

核心链路是：

- 请求进入过滤器链
- JWT 过滤器解析 `Authorization: Bearer <token>`
- 恢复当前登录用户和角色
- 放入 SecurityContext
- 再由配置决定这个接口是否允许访问

代码上主要集中在：

- `SecurityConfig`
- `JwtAuthenticationFilter`

### 4. Spring Security 里的认证和授权有什么区别？

- 认证：你是谁
- 授权：你能做什么

在你这个项目里：

- JWT 校验成功属于认证
- `CANDIDATE / HR / ADMIN` 角色访问不同接口属于授权

### 5. 过滤器链是什么？

Spring Security 会把请求依次交给一串过滤器处理。

这些过滤器会负责：

- 解析认证信息
- 建立安全上下文
- 拦截未授权请求
- 处理异常

你这里最重要的一层就是 JWT 过滤器，它把 token 里的用户信息恢复出来。

### 6. 你这个项目里为什么关闭了 Session，改成无状态？

因为项目采用的是 JWT。

JWT 更适合：

- 前后端分离
- REST API
- 无状态扩展

所以可以这样讲：

`我把安全会话配置成无状态，让后端不保存用户会话，请求只依赖 Bearer Token 完成身份恢复。`

### 7. 为什么有些接口要 `permitAll`？

因为不是所有接口都需要登录。

当前公开接口典型包括：

- 登录
- 注册
- 健康检查
- 岗位公开列表和详情
- Swagger 文档

其余需要身份的接口再走角色控制。

### 8. Spring Security 已经做角色控制了，为什么业务层还要再校验？

因为 Spring Security 更擅长“接口级访问控制”。

但业务层还要处理“数据级权限控制”，例如：

- 你虽然是 HR
- 但不代表你能操作所有 HR 的岗位

所以控制方式是两层：

- 安全框架管“能不能进接口”
- 业务层管“能不能操作这条数据”

### 9. 为什么项目里没有大量使用 `@PreAuthorize`？

这不是必须。

你当前项目主要通过：

- `SecurityConfig` 做路径级权限控制
- Service 做数据归属校验

这种方式也很常见，尤其在项目规模不大时，配置会更集中、更清楚。

### 10. Spring Security 的优势是什么？

- 和 Spring 生态集成深
- 支持过滤器链、安全上下文、认证授权模型
- 可和 JWT、OAuth2、多种认证方式结合
- 适合做统一的安全入口

### 11. Spring Security 的缺点是什么？

- 学习曲线比较陡
- 过滤器链和异常链路不熟时比较难排查
- 配置稍不小心就可能出现放行或拦截错误

### 12. 关于 Spring Security 的一句话总结怎么讲？

可以直接这样说：

`我在项目里用 Spring Security 统一承接 JWT 认证、接口放行规则和角色访问控制，再通过业务层补充数据归属校验，构成完整的安全访问链路。`

## MyBatis-Plus

### 1. MyBatis-Plus 是什么？

MyBatis-Plus 是 MyBatis 的增强工具，目标是减少重复 CRUD 代码，同时保留 MyBatis 的灵活性。

它常见能力包括：

- `BaseMapper`
- 条件构造器
- 分页插件
- 常见 CRUD 封装

### 2. 为什么项目里用 MyBatis-Plus？

因为这个项目有很多典型中后台场景：

- 列表分页
- 条件筛选
- 简单 CRUD
- 多张业务表管理

如果全手写 SQL，成本更高；如果直接上 JPA，又会有另一套抽象和心智负担。

所以可以这样讲：

`我选 MyBatis-Plus，是因为它在保留 MyBatis 灵活性的同时，能明显减少分页、筛选和基础 CRUD 的样板代码。`

### 3. 你这个项目里 MyBatis-Plus 主要怎么用？

你项目里主要用到了这几类能力：

- `BaseMapper<T>` 做基础 CRUD
- `Wrappers.lambdaQuery()` 构造查询条件
- `Page<T>` 做分页
- 插件方式接分页拦截器

这套能力在岗位、投递、面试、通知、管理员列表里都反复用到了。

### 4. 为什么用 `lambdaQuery`，而不是直接写字符串字段名？

因为 `lambdaQuery` 更安全，也更利于重构。

例如：

- 写 `JobEntity::getStatus`
- 比写 `"status"` 更不容易出错
- IDE 重构时也更稳

这也是你之前清理 type-safe warning 时的一部分工程考虑。

### 5. 你这个项目里分页是怎么做的？

做法很典型：

- 接口接收 `page` 和 `size`
- Service 里创建 `Page<T>`
- Mapper 调 `selectPage(pageRequest, query)`
- 最后统一组装成项目自己的 `PageResponse`

这样前端拿到的数据格式会更统一。

### 6. MyBatis-Plus 的优势是什么？

- 减少样板 CRUD
- 列表查询和分页写起来快
- 保留 MyBatis 风格，灵活度高
- 对中后台项目很实用

### 7. MyBatis-Plus 的缺点是什么？

- 复杂 SQL 最后还是要回到手写
- 如果过度依赖封装，容易忽略底层 SQL 性能
- 条件构造器写复杂后，可读性未必一直好

### 8. 为什么不是直接用 JPA？

可以这样答：

- JPA 更偏对象关系映射和实体生命周期管理
- MyBatis-Plus 更贴近 SQL 风格
- 当前项目大量是中后台查询、筛选、分页，MyBatis-Plus 更直接

一句话版：

`JPA 更偏 ORM 抽象，MyBatis-Plus 更偏轻量增强版 MyBatis；对这个项目，我更看重查询控制感和开发效率。`

### 9. MyBatis-Plus 适合什么场景，不适合什么场景？

适合：

- 中后台 CRUD
- 分页列表
- 条件筛选
- 结构相对清晰的业务表

不太适合直接靠它解决：

- 特别复杂的统计 SQL
- 多表复杂报表
- 深度定制化的数据访问逻辑

这些场景通常要补手写 SQL 或单独优化。

### 10. 关于 MyBatis-Plus 的一句话总结怎么讲？

可以直接这样说：

`我在项目里用 MyBatis-Plus 承接大部分列表、筛选和分页查询，通过 BaseMapper、lambdaQuery 和分页插件减少样板代码，同时保留了对 SQL 结构和业务查询的控制力。`

## MySQL / 数据库设计

### 1. 为什么项目主数据库选 MySQL？

因为这个项目是典型业务系统，核心关注点是：

- 持久化
- 事务
- 关系建模
- 一致性

MySQL 很适合承接这类招聘业务数据。

### 2. 你这个项目的核心表有哪些？

当前核心表主要是：

- `users`
- `roles`
- `user_roles`
- `jobs`
- `applications`
- `interviews`
- `notifications`

增强表还有：

- `operation_logs`

  用来支撑 RabbitMQ 消费幂等

### 5. 为什么 applications 要单独建表，而不是直接挂在 jobs 或 users 上？

### 6. 为什么 interview 要独立成表？

因为面试不是投递表上的一个简单字段。

它还有自己的：

- 面试时间
- 地点
- 链接
- 状态
- 结果
- 备注

而且后面如果要扩成多轮面试，也更容易演进。

### 7. 为什么 notification 也要落表？

因为通知不仅仅是“临时提示”，它还是用户可以查看的业务记录。

落表之后就能支持：

- 未读统计
- 已读管理
- 历史消息回看
- RabbitMQ 消费结果持久化

### 8. 你这个项目里索引是怎么考虑的？

主要思路是：

- 对高频筛选字段建索引
- 对关联字段建索引
- 对列表查询常用条件建组合索引

例如：

- 岗位的状态、截止时间、城市、分类
- 投递的候选人、状态
- 通知的接收人、已读状态、创建时间

### 9. 为什么要保留外键？

外键的价值是保证引用关系不乱。

例如：

- 投递必须引用真实岗位和候选人
- 面试必须引用真实申请
- 通知必须引用真实接收人

对这种偏中后台、业务关系明确的项目来说，外键是能帮忙兜底的。

### 10. 状态字段为什么要设计成枚举值？

因为这些字段本质上是有限状态集合。

例如：

- 岗位：`OPEN / CLOSED / EXPIRED`
- 投递：`APPLIED / REVIEWING / INTERVIEW / OFFERED / REJECTED`
- 面试：`SCHEDULED / COMPLETED / CANCELLED`

这样设计更容易：

- 做约束
- 做状态流转
- 做统计聚合

### 11. 如果让你总结这个项目的数据库设计思路，你会怎么说？

可以这样讲：

`我在数据库设计上优先保证核心业务关系清晰、状态边界明确，并通过唯一约束、外键和索引去支撑防重复投递、单轮面试约束、通知幂等和后台高频查询。`

## 事务与一致性

### 1. 什么是事务？

事务就是把一组操作当成一个整体来执行，要么都成功，要么都失败。

它主要解决的是：

- 数据一致性
- 中间状态不可见
- 部分成功带来的脏数据问题

### 2. 你这个项目里哪些地方用了事务？

当前项目里比较典型的事务场景有：

- 候选人投递岗位
- HR 更新投递状态
- HR 安排面试
- HR 更新面试
- 通知消费落库

这些操作都不是简单单表改一行，而是会牵动多处状态变化。

### 3. 为什么这些地方必须加 `@Transactional`？

因为它们往往涉及多步写操作。

例如安排面试时：

- 新建面试记录
- 更新申请状态到 `INTERVIEW`
- 记录操作日志
- 触发后续通知事件

如果中间一步失败，没有事务就会出现只写了一半的数据。

### 4. 如果没有事务，会出现什么问题？

常见问题包括：

- 申请已经创建，但通知没发
- 面试记录创建了，但申请状态没更新
- 状态更新了，但操作日志没记上
- 主业务失败了，部分脏数据却留在库里

这些都会让系统状态变得不可信。

### 5. 你这个项目里 RabbitMQ 和事务是怎么配合的？

这是很值得讲的一点。

当前做法是：

- 先完成本地数据库事务
- 事务提交成功后，再发布 RabbitMQ 消息

这样可以避免：

- 业务回滚了
- 但通知消息已经发出去了

这就是你现在项目里用 `TransactionSynchronizationManager` 处理的核心原因。

### 6. 为什么不在事务里直接发 MQ？

因为如果数据库最后回滚，而消息已经发出去，就会出现系统不一致：

- 库里没有这次状态变化
- 但消费者已经按这条消息做了通知

所以更稳的方式是“提交后发消息”。

### 7. 这样做能保证强一致吗？

不能完全保证分布式强一致。

它更接近一种工程上常见的平衡方案：

- 主业务先保证本地事务一致
- 消息和下游通知走最终一致

对于通知这类非强同步主链，这是合理的。

### 8. 什么叫最终一致性？

就是某个瞬间数据可能还没完全同步完成，但经过一小段时间后会收敛到正确状态。

在你这个项目里：

- 投递状态更新先成功
- 通知可能晚几百毫秒到几秒落库
- 但最终候选人还是能看到正确通知

### 9. RabbitMQ 消费端怎么避免重复写通知？

这就是幂等问题。

你现在的做法是：

- 给通知事件生成 `event_key`
- 在 `notifications` 表对 `event_key` 加唯一约束
- 消费前先查，插入时也有唯一约束兜底

这样即使消息重复投递，也不会重复生成通知记录。

### 10. Redis 缓存和数据库之间怎么保证一致性？

你当前项目采用的是比较经典的做法：

- 读时优先查缓存
- 写时先更新数据库
- 写完后主动失效相关缓存

这样做不是绝对强一致，但在业务里很实用。

### 11. 如果 RabbitMQ 或 Redis 出问题，主业务怎么办？

你这个项目的设计思路是：

- Redis 挂了，回退数据库查询，性能下降但主业务还能跑
- RabbitMQ 发布失败时，通知服务会降级直接落库

这说明你把中间件当成增强层，而不是唯一依赖层。

### 12. 关于事务与一致性的总结怎么讲？

可以直接这样说：

`我在项目里用本地事务保证岗位投递、状态流转、面试安排等核心写操作的一致性，再通过提交后发消息、通知幂等和缓存失效策略，在数据库、消息队列和缓存之间实现可控的最终一致。`
